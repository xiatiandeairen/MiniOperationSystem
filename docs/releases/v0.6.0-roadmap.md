# MiniOS v0.6.0 Roadmap — "Connect the Dots"

## 版本演进

```
v0.1.0  能跑        → 基础设施
v0.2.0  能教        → 操控命令
v0.3.0  能看见      → explain + /proc
v0.4.0  能感受      → compare + lab
v0.5.0  能破坏      → crash lab + 脚本
v0.6.0  能串联      → 管道 + 学习路径 + SDK 提取
```

## 核心洞察

v0.1–v0.5 每个版本增加了一类能力，但它们之间是离散的——用户做完 lab 后不知道下一步是 compare 还是 crash。同时，项目积累了多个可独立复用的组件（trace engine, MLFQ scheduler, bitmap allocator），是时候验证"SDK 拆分"的可行性。

## 三重视角

### 用户视角 — "给我一条学习路线"
- 当前 38 个命令太多，新用户不知道按什么顺序使用
- 需要一个**学习路径引擎**：根据已完成的步骤推荐下一步
- `journey` 命令：追踪用户进度，像游戏成就系统

### 专家视角 — "管道是 Unix 哲学的核心"
- `cat /proc/meminfo | grep Free` 是 Unix 最经典的组合
- 没有管道的 Shell 只是玩具，有管道才是工具
- 管道教学价值：进程间数据流、缓冲、stdout/stdin 概念

### 成熟软件视角 — "可复用组件是项目长期价值"
- minios-trace 作为独立 crate 可以用在嵌入式/no_std 场景
- MLFQ scheduler 可以用在任务调度库中
- Bitmap allocator 可以用在资源管理中

## 特性规划

### F1: 管道操作符 `|`

```
MiniOS $ cat /proc/meminfo | grep Free
MemFree:   63745 frames (254980 KiB)

MiniOS $ ps | head 2
PID  STATE    PRIORITY  CPU_TIME
0    READY    3         5000

MiniOS $ cat /proc/interrupts | wc
  2   10   65
```

实现：左侧命令的 println 输出捕获到 Vec<u8>，作为右侧命令的 stdin。
单级管道足够教学目的。

### F2: 学习路径引擎 — `journey` 命令

```
MiniOS $ journey
=== Your MiniOS Learning Journey ===

Chapter 1: Getting Started          [████████░░] 80%
  ✅ help — saw all commands
  ✅ ps — viewed processes
  ✅ meminfo — checked memory
  ⬜ ls / — browse filesystem

Chapter 2: Under the Hood           [██░░░░░░░░] 20%
  ✅ explain ls — learned VFS internals
  ⬜ trace follow ls / — traced a command
  ⬜ cat /proc/interrupts — saw interrupt stats
  ⬜ pagetable 0x4444_4444_0000 — walked page tables

Chapter 3: Experiments              [░░░░░░░░░░] 0%
  ⬜ lab scheduler-fairness
  ⬜ lab memory-usage
  ⬜ crash oom
  ⬜ compare scheduler

Next step: Try 'ls /' to browse the filesystem!
```

追踪方式：在每个命令的 handler 中调用 `journey::mark("command_name")`。

### F3: SDK 候选提取 — minios-trace 独立化

将 trace engine 拆分为可独立使用的 crate:
1. 移除对 `minios-hal` 的依赖（SerialJsonWriter 改为泛型 Write）
2. 添加 README 和完整 rustdoc
3. 添加独立的 example（不依赖 OS）
4. 测试覆盖 >80%

目标：验证"SDK 拆分"的可行性，为后续提取更多组件铺路。

### F4: Shell 补全提示

Tab 键触发命令名补全（简化版：前缀匹配）：

```
MiniOS $ com<TAB>
compare
MiniOS $ compare sch<TAB>
scheduler
```

### F5: `alias` 命令

```
MiniOS $ alias ll "ls /"
MiniOS $ ll
  d      0  dev
  d      0  proc
  ...
```

### F6: /proc 补充 — `/proc/self`

```
MiniOS $ cat /proc/self/pid
1
MiniOS $ cat /proc/self/name
init
```

## 优先级

| 优先级 | 特性 | 理由 |
|--------|------|------|
| P0 | F1 管道 | Unix 哲学核心，之前一直延后 |
| P0 | F2 journey | 解决"不知道下一步做什么"的根本问题 |
| P1 | F3 SDK 提取 | 验证长期价值方向 |
| P2 | F4 Tab 补全 | Shell 体验微改善 |
| P2 | F5 alias | 低成本，高实用 |
| P3 | F6 /proc/self | 小功能，教学补充 |

## 版本目标

v0.6.0 完成 F1 + F2 + F3（管道 + 学习路径 + SDK 提取验证），将离散的命令串联成完整的学习体验，同时验证项目的组件复用价值。
