# MiniOS v0.2.0 Roadmap — "Teach by Running"

## 版本定位

v0.1.0 证明了 MiniOS 可以引导和交互。v0.2.0 的目标是让它**真正教会开发者 OS 的核心机制**。

核心理念: 每个 OS 概念都应该有一个对应的 Shell 命令让用户亲手操作和观察。

---

## 方向一: 让调度可观测、可操控

**目标**: 用户能在 Shell 中创建任务、观察调度决策、理解时间片和优先级

### 特性

| 特性 | 说明 | 价值 |
|------|------|------|
| `spawn <name>` 命令 | 创建一个内核任务（如: 循环打印计数器） | 用户亲手创建进程 |
| `kill <pid>` 命令 | 终止一个进程 | 理解进程生命周期 |
| `sched` 命令 | 显示 MLFQ 各队列状态 + 当前时间片 | 透视调度器内部 |
| 抢占式调度 | Shell 作为 PID 1 任务参与调度（不再是 kernel_main 直连） | 理解抢占 |
| `nice <pid> <prio>` 命令 | 修改进程优先级 | 理解优先级影响 |

### 学习者体验
```
MiniOS $ spawn counter
Created task 'counter' with PID 2
MiniOS $ ps
PID  STATE    PRIORITY  CPU_TIME
0    READY    3         5000
1    RUNNING  0         4800
2    READY    0         200
MiniOS $ sched
Queue 0 [HIGH ]: PID 2
Queue 1 [MED  ]: (empty)
Queue 2 [LOW  ]: (empty)
Queue 3 [IDLE ]: PID 0
Running: PID 1, remaining: 1 tick
MiniOS $ kill 2
Process 2 terminated.
```

## 方向二: 让内存可见、可探索

**目标**: 用户能看到页表映射、帧分配状态、堆布局

### 特性

| 特性 | 说明 | 价值 |
|------|------|------|
| `pagetable <vaddr>` 命令 | 显示虚拟地址的 4 级页表遍历路径 | 理解地址翻译 |
| `frames` 命令 | 可视化帧分配 bitmap（用字符表示占用/空闲） | 理解物理内存布局 |
| `alloc <size>` 命令 | 手动分配堆内存并显示地址 | 理解堆分配器 |
| `/proc/pagetable` | 通过 cat 查看当前地址空间映射 | ProcFS 扩展 |

### 学习者体验
```
MiniOS $ pagetable 0x4444_4444_0000
  PML4[136] → PDPT @ 0x1234000
  PDPT[0]   → PD   @ 0x1235000
  PD[546]   → PT   @ 0x1236000
  PT[68]    → Frame 0x5678 (PRESENT | WRITABLE)
  Physical: 0x5678000
MiniOS $ frames
[####....####....####....] 67% used (42000/65456)
```

## 方向三: Trace 从功能到教学工具

**目标**: Trace 数据可以真正帮用户理解 "一次系统调用经过了哪些层"

### 特性

| 特性 | 说明 | 价值 |
|------|------|------|
| Syscall trace | 每个 syscall 自动产生 trace span | 可视化 syscall 路径 |
| `trace follow <cmd>` | 执行命令并只显示该命令产生的 trace | 聚焦单次操作 |
| Trace Viewer 实时加载 | 串口捕获 → 自动导入 Web viewer | 端到端数据通路 |
| 教学注解 | trace tree 输出附带 "这一步做了什么" 的解释 | 降低理解门槛 |

### 学习者体验
```
MiniOS $ trace follow cat /proc/meminfo
  [syscall] sys_open ("/proc/meminfo", READ)    0.3 μs
    [fs] vfs_open                                0.2 μs
      [fs] procfs_generate                       0.1 μs
  [syscall] sys_read (fd=3, 512 bytes)           0.5 μs
    [fs] vfs_read                                0.3 μs
      [fs] fd_table_read_virtual                 0.1 μs
  [syscall] sys_close (fd=3)                     0.1 μs
    [fs] vfs_close                               0.1 μs
Total: 4 spans, 0.9 μs
```

## 方向四: Syscall 真实陷入

**目标**: 实现 int 0x80 或 syscall 指令的真实系统调用机制

### 特性

| 特性 | 说明 | 价值 |
|------|------|------|
| `int 0x80` 陷入 | IDT 注册 0x80 handler, 从寄存器读取参数 | 理解特权级切换 |
| 用户态 wrapper | 提供 `syscall(num, arg1, arg2, arg3)` 内联汇编包装 | 模拟真实 libc |
| `strace <cmd>` | 实时跟踪命令的 syscall 调用序列 | 类似 Linux strace |

---

## 方向五: 独立 SDK 拆分（可复用组件）

**目标**: 将通用能力从 OS 内核中提取为独立 crate，可在其他项目中使用

### 候选 SDK

| SDK | 当前位置 | 独立价值 |
|-----|---------|---------|
| `minios-trace` | crates/trace | 嵌入式/no_std 环境的轻量级 trace 引擎 |
| `minios-mlfq` | crates/scheduler | 通用 MLFQ 调度算法实现 |
| `minios-ramfs` | crates/fs | 内存文件系统，适合测试和嵌入式 |
| `minios-framebuffer-console` | crates/hal/framebuffer.rs | 像素帧缓冲文本渲染 |
| `minios-bitmap-allocator` | crates/memory/frame.rs | Bitmap 帧分配器 |

### SDK 发布要求
- 独立 Cargo.toml，不依赖其他 minios crate
- 完整 rustdoc + README
- >80% 测试覆盖
- 发布到 crates.io

---

## 优先级排序

| 优先级 | 方向 | 理由 |
|--------|------|------|
| P0 | 方向一: 调度可观测 | 最高教学价值，用户能亲手操作进程 |
| P1 | 方向三: Trace 教学化 | 核心差异化特性，从"有" 到 "有用" |
| P2 | 方向二: 内存可见 | 页表是 OS 学习的重难点 |
| P3 | 方向五: SDK 拆分 | 项目长期价值，可独立产生影响力 |
| P4 | 方向四: Syscall 陷入 | 教学完整性，但实现复杂度高 |

## 版本目标

v0.2.0 完成 P0 + P1（调度可操控 + Trace 教学化），使 MiniOS 从 "能跑的 OS" 变成 "能教人 OS" 的工具。
