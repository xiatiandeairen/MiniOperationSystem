# MiniOS v0.5.0 Roadmap — "Break Things to Learn"

## 版本演进

```
v0.1.0  "Hello, MiniOS!"      → 能跑      (基础设施)
v0.2.0  "Teach by Running"    → 能教      (操控命令)
v0.3.0  "See the Machine"     → 能看见    (explain + /proc)
v0.4.0  "Feel the Difference" → 能感受    (compare + lab)
v0.5.0  "Break Things to Learn"→ 能破坏   (故障实验 + 脚本执行)
```

## 核心洞察

v0.1–v0.4 让用户能看、能操作、能比较。但真正深刻的理解来自**看到系统在边界条件下的行为**——OOM 时会怎样？无限递归会怎样？fork bomb 会怎样？同时，用户需要从"一次一条命令"进化到"编写脚本批量执行"，才能做更复杂的实验。

## 三重视角

### 用户视角 — "我想亲手让系统崩溃，然后理解为什么"
- 最深刻的学习发生在系统出错时
- 需要安全的"破坏→观察→恢复"循环
- 需要能批量执行命令来做复杂实验

### 专家视角 — "错误处理是 OS 最难教的部分"
- Page fault 是学生最困惑的概念 → 需要可控触发
- OOM 是真实系统最常见的故障 → 需要模拟体验
- 教学 OS 的优势是可以安全地演示灾难场景

### 成熟软件视角 — "Shell 脚本是运维的基础"
- Bash 脚本能力是 Unix 学习的核心
- MiniOS 的 Shell 需要最基础的脚本能力（顺序执行）
- /etc/init.d 启动脚本是理解引导过程的关键

## 特性规划

### F1: 故障实验室 — `crash` 命令

安全地触发内核故障场景，观察系统反应：

```
MiniOS $ crash list
Available crash scenarios:
  1. oom          — 耗尽堆内存，观察分配器行为
  2. stack        — 深度递归溢出栈，观察 page fault
  3. divide-zero  — 触发除零异常
  4. null-deref   — 解引用空指针
  5. fork-bomb    — 快速创建进程直到 max_processes

MiniOS $ crash oom
=== Crash Lab: Out of Memory ===
Allocating 1 KiB blocks until failure...
  Block 1: OK (heap free: 1047552)
  Block 2: OK (heap free: 1046528)
  ...
  Block 1020: OK (heap free: 1024)
  Block 1021: FAILED — alloc returned null

What happened:
  The linked-list allocator ran out of contiguous free blocks.
  In Linux, the OOM killer would terminate a process to free memory.
  MiniOS currently panics on OOM — this is a design limitation.

Recovery: All test allocations freed. System stable.
```

每个场景：触发 → 解释 → 安全恢复（不真正 panic）

### F2: 脚本执行 — `run` 命令

从文件系统读取并逐行执行命令：

```
MiniOS $ write /etc/demo.sh "ps\nmeminfo\nls /\ntrace stats"
MiniOS $ run /etc/demo.sh
> ps
PID  STATE    PRIORITY  CPU_TIME
0    READY    3         5000
1    RUNNING  0         4800
> meminfo
Frames: 63745 free / 65456 total
Heap:   16 used / 1048560 free
> ls /
  d      0  dev
  d      0  proc
  d      0  tmp
  d      0  etc
> trace stats
Total spans written: 150
```

### F3: 管道操作符 — `|`

Shell 支持单级管道，将前一个命令的输出作为后一个命令的输入：

```
MiniOS $ cat /proc/meminfo | head 2
MemTotal:  65456 frames (261824 KiB)
MemFree:   63745 frames (254980 KiB)
```

实现方式：将左侧命令的 println 输出捕获到内存缓冲区，然后作为右侧命令的输入。

### F4: head/tail/grep 文本工具

基础文本处理命令，让管道有意义：

```
MiniOS $ head 3 /proc/meminfo     → 显示前 3 行
MiniOS $ grep Free /proc/meminfo  → 过滤包含 "Free" 的行
MiniOS $ wc /proc/meminfo         → 统计行数/字数/字节数
```

### F5: 启动脚本 — `/etc/init.sh`

系统引导时自动执行 `/etc/init.sh` 中的命令：

```rust
// In kernel_main, after fs init:
if let Ok(script) = vfs.read_file("/etc/init.sh") {
    for line in script.lines() {
        shell::execute_line(line);
    }
}
```

默认 `/etc/init.sh`:
```
set VERSION 0.5.0
set PROMPT MiniOS
echo System initialization complete.
```

### F6: 测试覆盖率提升

为 v0.1–v0.4 中缺少测试的模块补充：
- Shell parser 测试（空行、多空格、引号边界）
- Environment variable 测试
- History ring buffer 测试
- compare/explain 输出不 panic 验证

## 不做的事
- 不做真实抢占式调度
- 不做磁盘持久化
- 不做网络
- 不做用户态隔离
- crash 命令不真正让系统 panic（只模拟并解释）

## 优先级

| 优先级 | 特性 | 理由 |
|--------|------|------|
| P0 | F1 crash 故障实验室 | 最深度的学习方式，其他 OS 教学工具没有 |
| P0 | F2 run 脚本执行 | 从交互式到自动化的关键跨越 |
| P1 | F4 head/grep/wc | 低成本文本工具，让 /proc 数据可过滤 |
| P2 | F5 /etc/init.sh | 理解引导过程，~20 行代码 |
| P2 | F6 测试覆盖 | 技术债务清理 |
| P3 | F3 管道 | 实现复杂度高，可延后到 v0.6.0 |

## 版本目标

v0.5.0 完成 F1 + F2 + F4 + F5（故障实验 + 脚本执行 + 文本工具 + 启动脚本），让用户从"安全操作"进阶到"破坏→理解→自动化"。管道(F3)延后到 v0.6.0。
