# MiniOS v0.4.0 Roadmap — "Feel the Difference"

## 版本演进回顾

```
v0.1.0  "Hello, MiniOS!"     → 能跑    (基础设施)
v0.2.0  "Teach by Running"   → 能教    (操控命令)
v0.3.0  "See the Machine"    → 能看见  (explain + /proc)
v0.4.0  "Feel the Difference"→ 能感受  (对比实验 + 动态演示)
```

## 核心洞察

v0.1–v0.3 已经让用户可以**看到**和**理解** OS 内部。但学习的深度来自**对比和实验**——不是告诉用户"MLFQ 有 4 个队列"，而是让用户亲手改变时间片长度、看到调度行为的变化，然后问自己"为什么 Linux 不用 MLFQ？"

## 三重视角分析

### 用户视角 — "我学完 explain 后下一步做什么？"
- 需要**实验式学习**：改参数→观察变化→理解 why
- 需要**对比参照**：MiniOS 的实现 vs 教科书/Linux 的做法
- 需要**成就感**：做完实验后有一个"你刚才学会了什么"的总结

### 专家视角 — "好的 OS 课程应该有哪些 lab？"
- MIT 6.S081 (xv6): syscall lab, page table lab, lock lab, file system lab
- 每个 lab 有明确的目标、验收标准和思考题
- MiniOS 可以做的：**内置 challenge 模式** —— Shell 中运行实验并自动验证

### 成熟软件视角 — "真实 OS 有什么 MiniOS 没有的？"
- **信号机制 (signals)** — 进程间异步通知 (SIGKILL, SIGSTOP)
- **管道 (pipe)** — `cmd1 | cmd2` 输出重定向
- **环境变量** — 简单的 key-value 配置
- 这些是"真实感"的来源，让学习者感受到"这真的是一个 OS"

## 特性规划

### F1: 对比学习模块 — `compare` 命令

让用户看到 MiniOS 的设计选择与行业标准的差异：

```
MiniOS $ compare scheduler
┌──────────────────┬──────────────────┬──────────────────┐
│ 维度             │ MiniOS           │ Linux            │
├──────────────────┼──────────────────┼──────────────────┤
│ 算法             │ MLFQ (4 级)      │ CFS (红黑树)     │
│ 时间片           │ 固定 2/4/8/16    │ 动态(vruntime)   │
│ 抢占             │ 协作式           │ 抢占式           │
│ SMP              │ 单核             │ 多核 per-CPU     │
│ 实时             │ 无               │ SCHED_FIFO/RR    │
└──────────────────┴──────────────────┴──────────────────┘

为什么 Linux 用 CFS 而非 MLFQ？
→ CFS 用虚拟运行时间保证公平性，不需要固定时间片。
→ MLFQ 需要调参（时间片大小、boost 间隔），CFS 自适应。
→ 但 MLFQ 更容易理解，所以是教学的好选择。
```

支持的对比主题: scheduler, memory, filesystem, ipc, syscall

### F2: 动态实验 — `lab` 命令

内置可验证的实验，引导用户深度理解：

```
MiniOS $ lab list
Available labs:
  1. scheduler-fairness  — 观察 MLFQ 的公平性
  2. memory-fragmentation — 观察堆碎片化
  3. page-table-walk     — 手动翻译虚拟地址
  4. ipc-latency         — 测量消息传递延迟
  5. trace-overhead      — 测量 trace 的性能开销

MiniOS $ lab scheduler-fairness
=== Lab: Scheduler Fairness ===
This lab creates 3 tasks and compares their CPU time.

Creating task A (priority HIGH)...  PID 2
Creating task B (priority MED)...   PID 3
Creating task C (priority LOW)...   PID 4

Waiting 200 ticks...

Results:
  PID 2 (HIGH): 95 ticks
  PID 3 (MED):  65 ticks
  PID 4 (LOW):  40 ticks

Question: Why does the HIGH priority task get more CPU time?
Answer: MLFQ gives shorter time slices to higher priorities,
but schedules them more frequently. Task A runs 2 ticks at a
time but gets picked first from Queue 0.

✅ Lab complete!
```

## 不做的事

- 不做真实抢占式调度 — cooperative 对教学足够，抢占式实现风险高
- 不做磁盘文件系统 — RamFS 足够教学，磁盘增加 10x 复杂度
- 不做网络栈 — 超出教学范围
- 不做 SDK 拆分 — 等功能稳定后再考虑

## 优先级排序

| 优先级 | 特性 | 理由 |
|--------|------|------|
| P0 | F1 compare | 对比学习是深度理解的关键路径 |
| P0 | F2 lab | 实验式学习是记忆最深的方式 |
| P1 | F6 history | 低成本, 大幅改善 Shell 体验 |
| P2 | F3 signal | 让 spawn/kill 更完整 |
| P3 | F5 env | 简单实现, 增加"真实 OS"感 |
| P4 | F4 pipe | 实现复杂度较高, 可延后 |

## 版本目标

v0.4.0 完成 F1 + F2 + F6（对比学习 + 动态实验 + Shell 历史），让用户从"看见机器"到"感受设计决策背后的 why"。
