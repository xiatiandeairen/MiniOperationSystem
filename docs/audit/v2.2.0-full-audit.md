## 三、质量债务

### 严重 🔴

1. **Shell 零测试**: 4,701 行代码、65 个命令，没有任何自动化测试。Shell 是用户直接交互的层，任何 bug 都直接影响体验。

2. **160 个空壳文档**: `docs/` 目录下 160 个 markdown 只有 1-2 行标题。这是批量生成的结果，不是真实文档。

3. **v2.0+ 功能未经交互验证**: v2.0.0 的"真实多任务"只通过串口日志验证了 Shell 启动，没有验证键盘输入、命令执行、spawn 任务可见运行。

### 中等 🟡

4. **basic.rs 膨胀**: 584 行、24 个命令塞在一个文件里。违反 plan.md 的 "文件 ≤500 行" 规则。

5. **空壳 crate**: `interrupt/` 只有 1 行，IDT 实际在 `hal/interrupts.rs`。

6. **架构偏离**: spec.md 定义了 `KernelServices` 注册表和多个 trait，实际代码从未使用 trait 多态——都是直接调用具体类型。

7. **VGA text mode 残留**: `hal/vga.rs` 的 VGA text buffer 已被 framebuffer 取代，但代码仍在。

### 轻微 ⚪

8. **Buddy 分配器是模拟**: 不在真实帧分配路径中，只在 lab 中使用。
9. **调度算法未接入内核**: FIFO/Priority/RR 只在 lab 模拟中使用，真正的调度器始终是 MLFQ。

## 五、未验证的假设

1. 65 个命令对学习者太多？可能只需要核心 15 个。
2. `explain` 的纯文本解释是否足够深？没有用户数据。
3. 启动 15 秒的等待是否可接受？
4. 帧缓冲字体（8×16）在高分屏上是否可读？

## 六、推荐的下一步（按 ROI 排序）

### 🥇 P0: 清理债务（做完才有基础做其他事）

| 任务 | 工作量 | 影响 |
|------|--------|------|
| 删除 160 个空壳文档 | 5 分钟 | 仓库干净 |
| 拆分 basic.rs (584行→3个文件) | 15 分钟 | 符合质量标准 |

| 标记 VGA text mode 为 deprecated | 5 分钟 | 明确代码状态 |

### 🥈 P1: 端到端验证（没有验证的功能等于不存在）

| 任务 | 工作量 | 影响 |
|------|--------|------|
| QEMU GUI 验证 v2.0 所有新命令 | 30 分钟 | 确认产品可用 |

### 🥉 P2: 选择一个战略方向深入

**方向 A: Web 化** — 将教学模拟编译为 WASM，浏览器直接运行
- 投入: 大 (新项目)
- 收益: 用户量 10x，零安装门槛
- 适合: 如果目标是 "让更多人学 OS"

**方向 B: 内核深化** — 真实抢占调度 + 可见的并发进程
- 投入: 中 (高风险)
- 收益: 教学深度质变
- 适合: 如果目标是 "教真正的 OS 内核机制"

**方向 C: 社区化** — 完善文档 + 贡献指南 + 开源推广
- 投入: 小
- 收益: 可能吸引贡献者
- 适合: 如果目标是 "建立社区"

## 七、诚实结论

MiniOS 在**广度**上已经做得很好——65 个命令、4 种调度算法、5 个 SDK 级 crate。但在**深度**和**质量**上有明显欠债：

1. Shell 代码占 37% 但零测试
2. 160 个空文档是快速迭代的副产品
3. 多个"实现了"的功能从未在真实环境中验证
4. 架构 spec 和实际代码已显著偏离

**建议**: 先做 P0（30 分钟清理），再做 P1（QEMU 验证），然后根据项目定位选择 P2 的方向。不要再加新功能，直到现有功能被验证为真正工作。
